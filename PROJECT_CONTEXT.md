## 24. 旧线程补充：Gradio 启动兼容与工具环境限制

旧线程中曾长期排查 Gradio 服务启动后退出、`ERR_CONNECTION_REFUSED`、localhost 自检失败等问题。

关键判断：这些问题不完全是项目业务代码错误，而是混合了以下因素：

- Gradio 会访问自己的 `/startup-events` 或用 `networking.url_ok(...)` 做 localhost 自检。
- 在代理环境或本机自检异常时，Gradio 可能主动抛错退出，并提示需要 `share=True` 或修正 localhost 访问。
- 项目只需要本地软著 demo，不应为了绕过自检而开公网 `share=True`。
- PowerShell 环境中出现过 `Path/PATH` 重复导致 `Start-Process` 不稳定。
- Codex 工具环境会回收后台子进程，因此旧线程里无法可靠地长期挂住 7860 服务。
- SQLite 写 `data/data.db` 时曾在工具沙箱/OneDrive 文件锁环境下被判定为只读，但这不一定代表用户自己终端启动会失败。

旧线程中的修复方向：

- 在 `run_oldcheat.py` 中设置 `NO_PROXY=127.0.0.1,localhost` 和 `no_proxy=127.0.0.1,localhost`。
- 关闭 Gradio analytics，减少代理环境下的超时噪音。
- 在启动器中跳过或覆盖 Gradio localhost 自检，使本地 demo 不因自检误判退出。
- 优先使用项目根目录下的 `run_oldcheat.py` 启动，而不是直接运行 `app/main.py`。

当前迁移后的判断：

- 现在新线程绑定到正确项目目录，项目本体读写不应再被旧工作区边界卡住。
- 如果仍出现启动问题，应优先检查端口占用、是否走 `run_oldcheat.py`、代理环境、以及是否有旧 Python 进程残留。

## 33. 旧线程补充：PDF 空壳修复与清洗数据用途
